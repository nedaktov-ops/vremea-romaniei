# VremeaRomaniei v1.1 ‚Äî Design Document

## Summary

This release focuses on hardening the existing codebase: removing dead code, fixing critical bugs, unifying fragmented color definitions, adding MapLibre error handling, and enabling the user's language preference.

## Changes

### 1. Dead Code Removal

**Unused DAOs and Entities:**
- `AlertDao.kt`, `LocationDao.kt`, `AlertEntity.kt`, `LocationEntity.kt` ‚Äî 0 callers across the entire codebase
- Removed files; `AppDatabase` now only registers `WeatherEntity` and `WeatherDao`
- Database version bumped from 1 to 2 (safe due to `fallbackToDestructiveMigration()`)

**Unused DataStore Preferences:**
- `windUnit` ‚Äî stored but never read or displayed
- `radarLayer` + `updateRadarLayer()` ‚Äî stored and defined but never invoked
- Removed from `PreferencesDataStore.AppSettings` and companion object keys

### 2. Alert Color Unification

**Problem:** Alert severity colors defined in 4 separate places with inconsistent values (MINOR was green in UI but blue in data; EXTREME was #D32F2F in Color.kt but #F44336 elsewhere).

**Fix:**
- Added `AlertSeverity.toHexColor(): String` and `AlertSeverity.toNotificationColor(): Int` on the domain model enum (single source of truth)
- `AlertRepository` now calls `severity.toHexColor()` instead of inline `when` blocks
- `AlertNotificationManager` now calls `alert.severity.toNotificationColor()` instead of inline `when`
- `Color.kt`: Added `AlertBlue = Color(0xFF2196F3)`; fixed `AlertRed = Color(0xFFF44336)` to match canonical value
- `AlertBanner`: Uses `AlertBlue` for MINOR (was `MaterialTheme.colorScheme.tertiary` ‚Äî green, inconsistent); uses canonical `AlertRed`

### 3. MapLibre Style Load Error Handling

**Problem:** `MapScreen` called `mapLibreMap.setStyle(...)` with zero error handling; a blank map on tile server failure with only a logcat message.

**Fix:**
- Registered `MapView.OnDidFailLoadingMapListener` on the `MapView` (not `MapLibreMap`)
- On failure: a `Card` overlay with `errorContainer` color, the error message, and a Retry button
- Retry clears the error and re-issues `setStyle()`
- Added `map_load_error` string resource in both EN (`"Failed to load the weather map..."`) and RO (`"Harta meteo nu s-a putut √ÆncƒÉrca..."`)

### 4. Language Preference Applied to Activity (CRITICAL BUGFIX)

**Problem:** The `language` setting was persisted in `PreferencesDataStore` and changeable in Settings screen, but `MainActivity` never read it nor applied locale configuration. Changing language had zero effect.

**Fix:**
- `MainActivity.attachBaseContext()` now reads the saved language from DataStore synchronously via `runBlocking` (DataStore file is <1KB; read is ~1ms) and applies locale via `createConfigurationContext()`
- `Locale.setDefault()` is also called so `java.util.Locale.getDefault()` picks up the change
- `lifecycleScope` observes the settings flow and calls `recreate()` when language changes
- **Bonus:** `WeatherCode.getDescription()` now defaults `ro` to `Locale.getDefault().language == "ro"`, so weather descriptions automatically follow the language setting

### 5. Room Schema Export (deferred)

**Attempted:** Set `exportSchema = true` + added `ksp { arg("room.schemaLocation", "$projectDir/schemas") }`.

**Blocked by:** `Room 2.8.4` internal `FieldBundle$$serializer` is incompatible with `kotlinx-serialization 1.8.0` (abstract method error). Reverted to `exportSchema = false` with a comment. Needs either Room upgrade or kotlinx-serialization downgrade.

### 6. Weather Code Auto-Detect Locale

`WeatherCode.getDescription(code)` now defaults `ro` to `Locale.getDefault().language == "ro"`, which means after `MainActivity` sets `Locale.setDefault()`, all weather descriptions automatically use the correct language.
